## Conventions

| Pattern | Behaviour |
|---|---|
| **`instance` is always the first argument** | Every operation receives the per-request lifecycle object returned by `Lib.Instance.initialize()`. The module reads `instance.time` for timestamps and uses `instance.backgroundRoutine` for the post-verify record delete |
| **Programmer errors throw `TypeError` synchronously** | Missing required option, wrong type, non-positive length / TTL throw `TypeError` at the call-site |
| **Operational errors return `{ success: false, error }`** | Cooldown rejection, not-found, expiry, max-fail lockout, wrong value, and store driver failures all come through the response envelope |
| **One `(scope, key)` holds at most one active record** | Calling `createPin` / `createCode` / `createToken` with the same `(scope, key)` replaces the previous record. There is no accumulation |
| **Success is strictly one-time** | A successful `verify` schedules a background `deleteRecord`. The same code submitted again finds nothing (`NOT_FOUND`). See [Runtime](runtime.md) for the serverless caveat |

---

## The Response Envelope

| Field | Type | Description |
|---|---|---|
| `success` | `boolean` | `true` on success. `false` on operational failure |
| `error` | `object \| null` | `{ type, message }` on failure. `null` on success. See [Error Catalog](#error-catalog) |
| `code` | `string` | The generated value (on `create*` success only) |
| `expires_at` | `number` | Unix epoch seconds when the code becomes invalid (on `create*` success only) |
| `deleted_count` | `number` | Number of expired records removed (on `cleanupExpiredRecords`) |

## Create Functions

Three generators share one underlying flow. The difference is the charset.

### `createPin(instance, options)` *(async)*

Generates a numeric code from `CONFIG.PIN_CHARSET` (default `0-9`). Common use case: SMS one-time password, phone-keypad entry.

### `createCode(instance, options)` *(async)*

Generates an alphanumeric code from `CONFIG.CODE_CHARSET` (default: Crockford Base32, `0-9 A-Z` minus `I L O U`). Common use case: codes that humans read aloud or printed. The charset deliberately avoids visually ambiguous characters.

### `createToken(instance, options)` *(async)*

Generates a URL-safe token from `CONFIG.TOKEN_CHARSET` (default `a-z A-Z 0-9`). Common use case: magic-link query parameter, click-to-verify email link.

**Options (identical across all three).**

| Option | Type | Required | Description |
|---|---|---|---|
| `scope` | `string` | Yes | Logical owner namespace. Part of the composite primary key |
| `key` | `string` | Yes | Specific verification purpose. Part of the composite primary key |
| `length` | `integer` | Yes | Number of characters in the generated code (must be `> 0`) |
| `ttl_seconds` | `integer` | Yes | Lifetime before the code becomes invalid (must be `> 0`) |
| `cooldown_seconds` | `integer` | Yes | Minimum gap before another `create*` call succeeds for the same `(scope, key)`. Use `0` to disable cooldown (must be `>= 0`) |

**Return shape.**

```js
// Success
{ success: true, code: 'X7K3M9', expires_at: 1730000300, error: null }

// Failure
{ success: false, code: null, expires_at: null, error: { type: 'VERIFY_COOLDOWN_ACTIVE', message: '...' } }
```

**Lifecycle (every `create*` call):**

1. Validate options (throws `TypeError` on programmer error).
2. `getRecord(scope, key)`. Store failure becomes `SERVICE_UNAVAILABLE`.
3. If a record exists and `instance.time - record.created_at < cooldown_seconds`, return `COOLDOWN_ACTIVE`.
4. Generate the code via `Lib.Crypto.generateRandomString(charset, length)`.
5. `setRecord(scope, key, { code, fail_count: 0, created_at, expires_at })`. Store failure becomes `SERVICE_UNAVAILABLE`.
6. Return `{ success: true, code, expires_at, error: null }`.

## Verify Function

### `verify(instance, options)` *(async)*

Consumes any code generated by `createPin`, `createCode`, or `createToken`. The same function handles all three; the module does not distinguish the type of code at verify time.

| Option | Type | Required | Description |
|---|---|---|---|
| `scope` | `string` | Yes | Same scope used at creation |
| `key` | `string` | Yes | Same key used at creation |
| `value` | `string` | Yes | Value the caller submitted (non-empty) |
| `max_fail_count` | `integer` | Yes | Number of failed attempts allowed before the record is locked out (must be `> 0`) |

**Return shape.**

```js
// Success
{ success: true, error: null }

// Failure
{ success: false, error: { type: 'VERIFY_WRONG_VALUE', message: '...' } }
```

**Lifecycle:**

1. Validate options (throws `TypeError` on programmer error).
2. `getRecord(scope, key)`. Store failure becomes `SERVICE_UNAVAILABLE`.
3. `record == null` returns `NOT_FOUND`.
4. `instance.time > record.expires_at` returns `EXPIRED`.
5. `record.fail_count >= max_fail_count` returns `MAX_FAILS`.
6. `record.code !== value` calls `incrementFailCount` (best-effort) and returns `WRONG_VALUE`.
7. On match, schedules `deleteRecord` via `Lib.Instance.backgroundRoutine` and returns `{ success: true, error: null }`.

The background delete is what enforces the one-time guarantee. In a serverless runtime, the container may freeze before the delete completes; see [Runtime](runtime.md) for the caveat.

## Operational Functions

### `setupNewStore(instance)` *(async)*

Idempotent backend setup. The actual work depends on the chosen storage adapter. SQL adapters run `CREATE TABLE IF NOT EXISTS` plus index DDL. NoSQL adapters may provision indexes or rely on out-of-band IaC. See your adapter's README for the exact behaviour.

**Return shape.** `{ success, error }`.

### `cleanupExpiredRecords(instance)` *(async)*

Bulk-delete records whose `expires_at` is in the past. Adapters with native TTL (MongoDB, DynamoDB) handle expiry automatically; this function is the explicit fallback. The verify module **never depends on cleanup running** for correctness: the consume-time `instance.time > record.expires_at` check guarantees expired codes are rejected even when the sweep is delayed.

**Return shape.** `{ success, deleted_count, error }`.

## Error Catalog

All operational errors live in `verify.errors.js`. Every failure path returns a frozen `{ type, message }` object from this catalog.

| `error.type` | Trigger |
|---|---|
| `VERIFY_COOLDOWN_ACTIVE` | `create*` called while a previous code's cooldown window has not elapsed |
| `VERIFY_NOT_FOUND` | `verify` with no record for the `(scope, key)` pair |
| `VERIFY_EXPIRED` | `verify` on a record past `expires_at` |
| `VERIFY_MAX_FAILS` | `verify` after `record.fail_count` has reached `max_fail_count` |
| `VERIFY_WRONG_VALUE` | `verify` on a record whose stored code does not match the submitted value |
| `VERIFY_SERVICE_UNAVAILABLE` | Any store adapter call returned `success: false` |

Error shape is frozen at module load:

```js
{ type: 'VERIFY_WRONG_VALUE', message: 'Verification code does not match' }
```

Projects may pass these through directly to clients, or map `error.type` to domain-specific errors in the service layer. No per-error `if`/`switch` is required.
